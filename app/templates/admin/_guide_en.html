1. Overview

{{ brand }} is a multi-tenant network monitoring & security platform. Lightweight probes deployed inside customer networks discover assets, scan for vulnerabilities, run intrusion detection (Suricata) and scan wireless (WiFi/BLE). Everything is reported to this central Command Center over an end-to-end encrypted channel.

How it fits together
  • Server (this app) — runs serverless (Vercel) on a PostgreSQL database. Manages tenants, probes, data and remote commands.
  • Probe — a containerized agent in the customer network. Pairs with a License Code, then is fully driven from the server.
  • Encryption — X25519 key exchange + AES-256-GCM for all probe⇄server data.

2. Getting Started

You sign in with your email + password. There are two roles:

  • Superadmin — sees and manages everything across all tenants, plus System pages (Settings, IDS Rules, System Health).
  • Tenant admin — sees only their own tenant's probes, assets and threats.
First-time setup (superadmin)
1
Go to Tenants & Licenses → create a Tenant (the customer).
2
Generate one or more License Codes for that tenant (one per probe).
3
Optionally create a Tenant Admin user (email + password) for the customer.

3. Deploy & Pair a Probe

1
Run the probe container on a machine inside the target network (docker compose up -d). It exposes a local UI on port 5001.
2
Open the probe UI → Dashboard → fill the registration form: probe name, optional location/contact, Tenant name and License Code.
3
Click Connect. The probe performs the encrypted handshake and becomes Paired. The green LED in the probe top bar confirms it is online with the server.
4
The probe now appears under Probe Fleet on the server and starts auto-scanning the network every 30 min.

For WiFi/BLE and Suricata the probe container needs network/Bluetooth privileges (provided in the bundled docker-compose.yml).

4. Dashboard & Tenant Context

The Dashboard shows live KPIs (probes online, assets, open services, vulnerable assets, data ingested) and a geographic map of your probes.

Active tenant selector (top bar, superadmin): pick a tenant to scope every page to it, or "All Tenants". Tenant admins are always scoped to their own tenant, shown as a red badge.

5. Probe Fleet & Remote Control

Probe Fleet lists every probe with status, reachability, asset count, IDS state and a quick Suricata Start/Stop. Click the Console button to fully pilot a probe.

Probe Console — available commands
  • scan_network — full network discovery
  • vuln_scan — vulnerability scan of an IP
  • scan_wifi / scan_ble — wireless scans
  • suricata_start/stop — IDS control
  • suricata_reload_rules — re-sync ruleset
  • set_scan_config — set subnet
  • get_logs / get_status — diagnostics
  • capture_pcap — capture N packets
  • self_update / restart_agent
  • factory_reset — wipe pairing (signed)

Commands are queued and picked up within seconds (the heartbeat speeds up while commands are pending). The command timeline shows each command's status and result. Sensitive commands are cryptographically signed. Use the Wireless page's fleet buttons to command all probes at once.

6. Network Assets & Vulnerabilities

Network Assets lists every discovered device: IP, MAC, OS, hostname, SNMP, open ports/services.

  • Scan — runs a Nmap vulnerability scan (--script vuln) on that device.
  • Vulns → modal with three tabs:
    • Vulnerability Data — raw scan output.
    • AI Analysis — a Groq-powered security report (needs a Groq key in Settings).
    • CVE Intelligence — live NVD enrichment for each CVE, with a self-refreshing cache.

7. Suricata IDS & Rules

Start/stop Suricata per probe from the Fleet or Console (an interface of "auto" selects the best physical NIC). Alerts stream back to the Threat Monitor.

IDS Rules (superadmin)
1
Download & Prepare a ruleset from a URL (default: Emerging Threats Open). Supports .rules and .tar.gz.
2
Pick a Detection Template (Malware & C2, Network Recon, Web Attacks, Exploits, Policy, or Full Coverage) to instantly enable matching categories.
3
Fine-tune Active Rule Categories with the checkboxes. Probes download only the active subset on their next Suricata start.

8. Threat Monitor & Threat Intelligence

  • Threat Monitor — a live terminal feed of IDS events (alerts, flows, DNS, TLS) streamed from the fleet.
  • Threat Intelligence — aggregated analytics: top signatures, top source/destination IPs, severity breakdown, plus an AI Threat Briefing (Groq) that interprets the telemetry into actionable intelligence.

9. Wireless (WiFi / BLE)

The Wireless page shows discovered WiFi access points (SSID, channel, signal, security) and BLE devices (name, address). Trigger scans on the whole fleet with the Scan WiFi / Scan BLE buttons, or per-probe from the Console. Results are deduplicated and stored like node discovery. Probes can also scan locally from their own Wireless page.

10. Notifications & Reports

  • Telegram & Gmail — configure a bot token and a Gmail address + App Password under Settings → Notifications, then use the test buttons.
  • PDF Reports — from Billing, download a per-tenant security report (fleet, assets, alerts, billing) or email it as an attachment.

11. Billing & Accountability

Each tenant has a fixed fee plus pay-per-use pricing (per probe, asset, scan, IDS alert). The Billing page computes current usage and the estimated total from real data; superadmins can edit prices and currency per tenant.

12. Settings (superadmin)

Organized into tabs: AI Analysis (Groq + alternative providers), Vulnerability Intel (NVD key, CVE cache validity), and Notifications (Telegram/Gmail + tests). System Health shows DB status, runtime versions and record counts.

13. Security Model

  • Pairing — X25519 ECDH handshake with a challenge; one-time License Codes.
  • Data channel — AES-256-GCM with a per-probe session key (scan data, IDS events, wireless results).
  • Commands — sensitive commands (factory reset, restart, self-update) are HMAC-signed, time-limited and replay-protected.
  • Isolation — every page and query is scoped to the tenant.

14. Troubleshooting

Choose a valid interface or "auto". Check the probe Console timeline — the command result shows the exact reason (e.g. Suricata not installed, no privileges). The probe needs NET_ADMIN/NET_RAW.

Make sure a ruleset is prepared under IDS Rules with at least one active category, then start Suricata so the probe downloads the active rules. Flows/DNS appear even without rules; alerts require rules + matching traffic.

Commands are delivered on the next heartbeat (seconds). Open the probe Console to watch the command move pending → running → done, with its result. The probe must be online (green LED).

The probe host needs a Bluetooth adapter and the container needs access to it (privileged + host D-Bus, as in the bundled compose). WiFi needs a wireless interface and the iw tool.
